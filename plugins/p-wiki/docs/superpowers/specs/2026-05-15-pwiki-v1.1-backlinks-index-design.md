# Design: `pwiki` v1.1 — `backlinks` and `index` commands

**Date:** 2026-05-15
**Status:** Drafted (brainstorming)
**Targets:** `plugins/p-wiki` v1.0.0 → v1.1.0 (minor bump)
**Predecessor:** `2026-05-14-pwiki-cli-design.md` (v1 CLI design)
**Companion:** `2026-05-14-confluence-destination-sketch.md` (separate v2 release, not in this scope)

---

## 1. Goal

Replace the two remaining non-CLI mechanical steps in the `compile` skill with deterministic CLI commands, completing the v1 vision (§8 of the v1 design) for the FS destination:

- **Step 4f — Backlink audit** → `pwiki backlinks <path>`
- **Step 5 — `index.md` regeneration** → `pwiki index`

These were explicitly deferred from v1 because each has edge cases (case-sensitive whole-word matching, code-block skipping, per-file dedup, suspicion threshold for backlinks; group rendering, summary extraction, atomic write for index) that warranted a separate session.

### 1.1 Pains addressed

- **Token cost.** The current `compile` skill grep-scans every `pages/**/*.md` once per created/updated page during step 4f and once globally during step 5. Each Edit and Read consumes LLM tokens. Moving both to the CLI eliminates the remaining read+edit cycle from the compile token budget. This is the same rationale as v1 §1.1 — completing the work, not introducing a new principle.
- **Determinism.** The same wiki state must produce the same backlink insertions and the same `index.md` output, regardless of which Claude session runs `compile`.

### 1.2 Non-goals (out of scope for v1.1)

- **Confluence destination.** Tracked separately as v2 of the plugin (`v2.0.0`); see `2026-05-14-confluence-destination-sketch.md`. v1.1 stays FS-only and additive.
- **Batch `backlinks` API.** A future `pwiki backlinks --pages p1,p2,...` form is not in scope; v1.1 ships only the single-page form invoked in a loop.
- **Selective accept/reject of backlink candidates** above the suspicion threshold. v1.1 is binary: insert all (with `--force`) or none. Manual editing post-CLI is the escape hatch.
- **New lint checks** (e.g. `missing-backlinks` — pages that mention a concept's `title` without linking). Out of scope; existing `underlinked` check is unaffected.
- **Frontmatter schema changes.** No `description:` or `summary:` field is introduced; index extracts summary from body. See §3.2.

### 1.3 One architectural exception flagged

The v1 design states (`wiki-claude-md.template.md`, line 157): *"The CLI never touches body text."* `pwiki backlinks` violates this in a tightly scoped way (deterministic hyperlink insertion at one position per file, governed by strict skip-rules). The rule is updated, not removed — the CLI touches body **only** for `pwiki new` (template body of a new page) and `pwiki backlinks` (hyperlink insertion). See §4.4.

## 2. Architecture

### 2.1 Destination interface additions

Two methods are added to the `Destination` contract declared in v1 (`tools/lib/destination.mjs`):

```
Destination {
  // ... existing v1 methods ...

  applyBacklinks({ targetPath, maxSuggestions, force })
    → { target, title, inserted[], total }              // exit 0 shape
    | { target, title, suspicious: true, total, candidates[] }  // exit 2 shape

  regenerateIndex()
    → { path, groups: { concept, person, source, query }, written: true }
}
```

`applyBacklinks` operates on one target page. The FS implementation scans `pages/**/*.md` (excluding the target itself), finds eligible insertion sites for the target's `title:`, and either inserts (exit 0) or refuses with a candidate list (exit 2, suspicious).

`regenerateIndex` takes no arguments. It rebuilds `docs/wiki/index.md` from current frontmatter and writes the result atomically.

Both additions are additive to the v1 interface; no existing method changes. The Confluence destination (v2) will implement these methods with backend-specific semantics — that is precisely the seam v1 §2.6 reserved.

### 2.2 New FS-implementation modules

```
plugins/p-wiki/tools/lib/
├── backlinks.mjs            ← scan pages, match title, edit-in-place
├── index.mjs                ← collect pages, render markdown, atomic write
├── md.mjs                   ← `sanitize(text)` helper for bare <word> tokens
└── destinations/fs.mjs      ← + applyBacklinks + regenerateIndex, delegating to the above
```

Rationale for separate modules (approach A from brainstorming): symmetric with v1's `search.mjs` and `lint.mjs`; clean separation between mutation (`backlinks`) and read-only inspection (`lint`, `search`); easier per-feature testing.

### 2.3 Reuse of v1 modules

- `paths.mjs` — wiki root discovery (no change).
- `fm.mjs` — frontmatter read for both target and candidates; body extraction.
- `yaml.mjs` — used transitively via `fm.mjs`.
- `slug.mjs` — sort key for index groups (alphabetic by slug within group).

No new npm dependencies. Markdown parsing required by `backlinks.mjs` and `index.mjs` is a narrow subset (H1 detection, paragraph extraction, fenced/inline-code skipping, link skipping) implementable as a state machine in ≈150 lines total.

### 2.4 Schema source of truth

`schema.mjs` is unchanged. No frontmatter field added or modified.

## 3. Commands

### 3.1 `pwiki backlinks <path>`

Insert markdown links to `<path>` in pages under `pages/**/*.md` that mention `<path>`'s `title:` in their body. Deterministic; subject to strict skip-rules.

`<path>` is repo-root-relative POSIX (`docs/wiki/pages/concept/foo.md`) per v1's path conventions (§2.3 of v1 design).

| Option | Default | Notes |
|---|---|---|
| `--max-suggestions <N>` | `20` | Suspicion threshold. Exceeded → exit 2, no writes. |
| `--force` | off | Insert all matches regardless of count. |
| `--format <text\|json>` | `json` | Skills consume json; `text` for human reading. |

**Algorithm:**

1. Read `<path>`'s frontmatter. Extract `title`, then `.trim()`. If file unreadable, frontmatter unparseable, or trimmed `title` is empty → exit 1.
2. Glob `<root>/docs/wiki/pages/**/*.md`. Remove `<path>` from the list (skip-self).
3. For each candidate file F:
   - Read F. Separate frontmatter from body.
   - Walk body with a state machine tracking: inside fenced block (` ``` ` to ` ``` `), inside inline code (`` `…` ``), inside any markdown link form — `[text](url)` (inline), `[text][id]` (reference), or bare `[text]` (shortcut reference). All three forms count as "inside a link" and their contents are skipped.
   - Note on headings: lines starting with `#` are **not** treated specially — they are body text for the state machine. A title that appears inside a heading (e.g. `## Foo` where Foo is the target title) will be linked, producing `## [Foo](rel/path.md)`. This is valid markdown and matches current compile 4f behavior; if a user finds the result undesirable, they remove it via git.
   - Find the **first** occurrence of the title surrounded by non-letter / non-digit boundaries (or start/end of body). Boundary detection uses Unicode property escapes (`\p{L}`, `\p{N}`) — JS's native `\b` matches ASCII only, which is insufficient for a bilingual (English + Russian) wiki.
   - The title itself is regex-escaped before matching (it can contain `.`, `(`, etc.).
   - If found, record `(F, line, column, surrounding-text-excerpt)`.
4. After scanning all candidates:
   - If `candidates.length > maxSuggestions` and not `--force` → exit 2 with JSON (§3.1 output shape below). **No file is modified.**
   - Otherwise, for each candidate insert the link: replace the matched `title` with `[title](<rel-path-from-F-to-target>)`. Modify body only; frontmatter is untouched.
5. Write each modified file atomically: write to `<file>.tmp`, `fs.rename` to `<file>`. Files are written one at a time, so a mid-run interrupt leaves some candidates updated and some untouched — never half-written. Per-file dedup is implicit (only one match per file is collected in step 3).

**`updated:` is not bumped.** `pwiki backlinks` does not modify the frontmatter of any file it touches — only body. The `updated:` field reflects content changes, not derived-metadata changes like incoming link insertion. Otherwise every `compile` run would re-stamp `updated:` on half the wiki, defeating the field's purpose.

**Output (exit 0, success):**

```json
{
  "target": "docs/wiki/pages/concept/kafka-partitioning.md",
  "title": "Kafka partitioning",
  "inserted": [
    { "file": "docs/wiki/pages/concept/streaming-architectures.md", "line": 14 },
    { "file": "docs/wiki/pages/source/article-foo-summary.md", "line": 7 }
  ],
  "total": 2
}
```

**Output (exit 2, suspicious):**

```json
{
  "target": "docs/wiki/pages/concept/plan.md",
  "title": "Plan",
  "suspicious": true,
  "total": 47,
  "candidates": [
    { "file": "docs/wiki/pages/concept/streaming.md", "line": 12, "preview": "...we plan to..." },
    { "file": "docs/wiki/pages/concept/auth.md", "line": 3, "preview": "...a plan that fits..." }
  ]
}
```

Skill behavior on exit 2: show user the candidate list, prompt for `force` / `skip` / `raise threshold to N`, retry CLI with the chosen flag.

**Exit codes:**
- `0` — success (including zero insertions).
- `1` — target unreadable or has no `title` in frontmatter.
- `2` — suspicious count; no files modified.
- `3` — internal CLI error.

**Idempotency.** Re-running on a wiki where backlinks already exist is a no-op: the previously inserted text reads as `[title](rel/path.md)` — inside a markdown link — and is therefore skipped in step 3.

### 3.2 `pwiki index`

Regenerate `docs/wiki/index.md` from current frontmatter of all `pages/**/*.md`. Full rebuild; no incremental updates.

| Option | Default | Notes |
|---|---|---|
| `--format <text\|json>` | `json` | `text` → print the rendered markdown to stdout, **do not write** the file (acts as preview/dry-run). |

**Algorithm:**

1. Glob `<root>/docs/wiki/pages/**/*.md`. For each, read frontmatter; group by `type:` into 4 buckets: `concept`, `person`, `source`, `query`.
2. Within each group, sort alphabetically by `id:` (= slug).
3. For each page, compute summary:
   - Read body. Strip frontmatter.
   - Skip leading blank lines and any line beginning with `#`.
   - Take the first non-empty paragraph (lines until next blank line).
   - From that paragraph, take the first sentence: split on the first `[.!?]` followed by whitespace or end-of-paragraph.
   - If no sentence terminator found, truncate at 200 characters and append `…`.
   - If body is empty or contains only headings → summary is `_(no summary)_` (rendered literally).
   - Apply `md.sanitize()` to the summary.
4. Render the markdown output:
   ```markdown
   # Wiki index

   _Regenerated by `pwiki index`. Don't edit by hand._

   ## Concepts
   - [Title A](pages/concept/title-a.md) — Summary.
   - …

   ## People
   _(none yet)_

   ## Sources
   - …

   ## Queries
   - …
   ```
   - Empty group → `_(none yet)_` (rendered literally, no list).
   - Group headers are fixed: `## Concepts`, `## People`, `## Sources`, `## Queries`.
   - The list-item summary is rendered with em-dash separator: `- [<title>](<path>) — <summary>`. If summary is `_(no summary)_`, render `- [<title>](<path>) — _(no summary)_`.
5. With `--format=json` (default): write atomically (`index.md.tmp` → `fs.rename` → `index.md`).
6. With `--format=text`: print the rendered markdown to stdout. Do not write the file.

**Output (`--format=json`):**

```json
{
  "path": "docs/wiki/index.md",
  "groups": { "concept": 12, "person": 3, "source": 8, "query": 2 },
  "written": true
}
```

**Output (`--format=text`):** the rendered markdown body as plain stdout, no JSON wrapper.

**Exit codes:**
- `0` — success (including any group being empty).
- `1` — wiki root not found, or, in `--format=json` mode, write error (`fs.rename` fails, disk full, permissions). In `--format=text` mode there are no writes, so exit 1 occurs only on root-not-found.
- `3` — internal CLI error.

`pwiki index` never returns exit 2 — there is no caller-resolvable conflict surface.

## 4. Skill migration

### 4.1 `compile` skill

Two steps in `plugins/p-wiki/skills/compile/SKILL.md` are rewritten; the rest of the skill text is unchanged.

**Step 4f — Backlink audit (current Grep/Edit loop)** is replaced with:

The audit runs **once at the end of step 4** for the full set of pages touched across all sources processed in this compile run (not per-source). For each such page:

```
  node "${CLAUDE_PLUGIN_ROOT}/tools/pwiki.mjs" backlinks <page-path> --format=json

- exit 0: parse JSON, accumulate `inserted.length` into the run total.
- exit 2: read `candidates` from JSON. Show the user the first 10 candidates
  with their preview snippets, then ask: insert all (`--force`), skip this
  target, or raise threshold to a chosen N. Retry the CLI accordingly:
    - "insert all" → re-run with `--force`
    - "skip" → continue to the next target
    - "raise to N" → re-run with `--max-suggestions=N`
- exit 1: report the failed target in the final summary; continue with the
  next target (per-page failure does not abort compile).
- exit 3: forward stderr; abort compile and ask the user to file an issue.
```

**Step 5 — Regenerate `index.md` (current Glob/Read/Write block)** is replaced with:

```
After all backlink audits in step 4f complete, invoke:

  node "${CLAUDE_PLUGIN_ROOT}/tools/pwiki.mjs" index --format=json

- exit 0: include "N concept / M people / K sources / J queries pages indexed"
  in the final report.
- exit 1: forward stderr to the user with a note "compile completed but index
  was not regenerated; run `pwiki index` manually after resolving the error".
  Do not abort compile (the page work is already done).
- exit 3: forward stderr; flag as CLI bug.
```

**Step 6 — Report** is unchanged in structure. "K backlinks added" is the sum of `inserted.length` across all `pwiki backlinks` calls. "Any backlink-audit warnings deferred for human review" is the list of targets where the user selected `skip` on an exit-2 prompt.

**Markdown sanitization section** in `SKILL.md` is preserved for the body content the skill writes (page bodies, source summaries). The third use-case mentioned there — `index.md` summary lines — moves into the CLI (`md.sanitize` in `index.mjs`); the section text is updated to drop the index reference.

**`allowed-tools` for compile** stays the same: `Bash(git rev-parse:*) Bash(node:*) Read Write Edit Grep Glob`. `Grep` is retained because step 4c (existing-page lookup by normalised title) still uses it; everything else either already uses Bash(node:*) or doesn't change.

### 4.2 `wiki-claude-md.template.md`

Three updates:

1. **Compile rules section** — append the new commands to the "Backlink audit during compile is mandatory" bullet:
   > **Backlink audit during compile is mandatory** — performed via `pwiki backlinks <path>` for each created/updated page. **Index regeneration** — performed via `pwiki index` at the end of compile.

2. **CLI tool section** — append two bullets to the existing list:
   > - **Backlink audit** — `pwiki backlinks <path>` (inserts hyperlinks to `<path>` in other pages where its `title:` is mentioned; exit 2 if the count exceeds the suspicion threshold).
   > - **Index regeneration** — `pwiki index` (rewrites `docs/wiki/index.md` from frontmatter; `--format=text` prints to stdout without writing).

3. **CLI tool section, the line stating *"The CLI never touches body text"*** — rewrite to:
   > Generic Read/Write/Edit remain for **body editing** in skills (adding facts to sections, synthesizing answers, conflict callouts). The CLI touches body text only in two specific deterministic operations: rendering the template body of a new page (`pwiki new`) and inserting backlink hyperlinks (`pwiki backlinks`).

### 4.3 Other skills

`init`, `ingest`, `lint`, `query` — unchanged. None of them call backlinks or index.

The `lint` skill's `underlinked` check (concept page < 3 outgoing links) is semantically unaffected by `pwiki backlinks`: backlinks inserts links into **other** pages pointing to the target, not outgoing links of the target itself. No correction needed.

## 5. Error handling at the skill boundary

| Exit | Command | Skill response |
|---|---|---|
| 0 | `backlinks` | Parse JSON; add `inserted.length` to the run-total. Continue with next target. |
| 0 | `index` | Parse JSON; include group counts in the final report. |
| 1 | `backlinks` | Forward stderr identifying the failed target. **Do not abort compile** — per-page failure is isolated; record the target in the final "skipped" list and continue. |
| 1 | `index` | Forward stderr. Do not abort compile (page work is done). Advise running `pwiki index` manually after the user fixes the cause. |
| 2 | `backlinks` | Show user the candidates list with previews. Prompt for `force` / `skip` / `raise threshold to N`. Retry the CLI with the chosen flag. |
| 3 | both | Forward stderr; ask user to file an issue. Abort compile (undefined state). |

Principle: `backlinks` per-page failure is isolated and recoverable in the report; `index` is a single global artifact and its failure is surfaced but does not abort the compile run (since the page mutations of compile steps 1-4 are already on disk).

## 6. Testing

Three layers, all running under existing `npm test` / `npm run validate`.

### 6.1 Unit tests

**`backlinks.test.ts`** — pure logic, no filesystem:
- `findMatches(body, title)` honors all skip-rules: inside `[..](..)`, inside fenced ` ``` ` (multi-line), inside inline `` `..` ``, case-sensitive matching.
- Word-boundary detection uses Unicode property escapes — assert that `\p{L}\p{N}`-based boundaries match correctly for Cyrillic titles (`Кафка` is not matched inside `Кафкианский`) and for English titles (`Kafka` is not matched inside `Kafkaesque`).
- First-occurrence-only per file (per-file dedup).
- Empty body / body without the title → no matches.
- Title that contains special regex characters (e.g. `Node.js`, `C++`) is escaped before matching.
- Relative-path computation for varying depths (`pages/concept/x.md` → from `pages/source/y.md` resolves to `../concept/x.md`).
- Skip-self: when target path equals candidate path, candidate is excluded.

**`index.test.ts`** — pure logic, no filesystem:
- Summary extraction: first paragraph after H1, split on `[.!?]\s|$`, fallback truncation at 200 chars + `…`.
- Empty body / body containing only an H1 → `_(no summary)_`.
- `md.sanitize()` applied (bare `<tenant>` → `` `<tenant>` ``).
- Grouping by type; alphabetical sort by `id` within each group.
- Empty group → `_(none yet)_` (no list).
- Slug-based sort is stable when titles contain Cyrillic — since slugs are always ASCII per v1 schema, sort is plain lexicographic on `id` (no locale-aware comparison needed).
- Output rendering: full sample wiki → exact markdown matches a fixture.

**`md.test.ts`** — sanitize helper:
- Bare `<word>` → ``` `<word>` ```.
- Token inside inline code is left alone.
- Token inside a fenced block is left alone.
- Non-word characters between `<>` (e.g. `<3` or `< 1ms`) are not sanitized.

### 6.2 Destination contract / shape conformance

`destination-contract.test.ts` (existing v1 file) is extended:
- `applyBacklinks` and `regenerateIndex` are present on the resolved Destination.
- Argument-validation behaves uniformly (missing/invalid arguments produce a consistent error shape).
- Output JSON matches §3 schemas (success-path and exit-2 path for `backlinks`; success-path for `index`).

This is the layer that v2-Confluence will reuse.

### 6.3 Destination-fs semantic tests

**`destination-fs-backlinks.test.ts`** — synthetic wiki in `os.tmpdir()`:
- Pristine wiki, no other pages → `applyBacklinks` succeeds with 0 insertions.
- Wiki with N candidates under threshold → all matched, all written.
- Idempotency: re-run on the same wiki → 0 insertions (existing links are skipped per skip-rules).
- Suspicious case (`candidates.length > maxSuggestions`, `force=false`) → returns `{ suspicious: true, candidates }`; **no file on disk is modified** (asserted by hashing files before and after).
- `force=true` at suspicious count → all candidates inserted.
- Candidate file contains the title inside a fenced block → skipped.
- Candidate file contains the title inside an existing `[..](..)` link → skipped.
- Target page contains its own title → skipped (skip-self).

**`destination-fs-index.test.ts`** — synthetic wiki in `os.tmpdir()`:
- Mixed wiki (3 concept + 1 person + 2 source + 0 query) → output matches expected markdown fixture exactly.
- Empty wiki (no pages) → all four groups render `_(none yet)_`.
- `--format=text` → stdout contains the rendered markdown, `index.md` on disk is not created (assert with `fs.stat` throw).
- Page with body that is empty / only an H1 → summary is `_(no summary)_`.
- Page with body containing bare `<placeholder>` in the first sentence → summary is sanitized.

### 6.4 Integration test

`cli-integration.test.ts` (existing v1 file) is extended with a new sequence at the end:

```
1. (existing v1 e2e steps producing concept and query pages)
2. node pwiki.mjs backlinks pages/concept/<existing>.md → exit 0, 0 insertions
3. Create a second concept page whose body mentions the first concept's title
4. node pwiki.mjs backlinks pages/concept/<first>.md → exit 0, 1 insertion
5. Assert the second page's body now contains the link
6. node pwiki.mjs index → exit 0, docs/wiki/index.md exists
7. Read index.md → both concept pages appear in the Concepts section
8. node pwiki.mjs lint → exit 0, no errors (new backlinks did not break dead-links etc.)
9. node pwiki.mjs index --format=text → stdout has markdown, index.md unchanged
```

### 6.5 Snapshot fixtures

`index.mjs` output for a small reference wiki (3-4 pages across types) is captured as a snapshot fixture. Snapshot is byte-compared in test; any change in rendering produces an explicit diff in PR review.

## 7. Backwards compatibility

**No breaking changes.** All additions are surface-level extensions of v1:

- New CLI commands `backlinks` and `index` — additive; existing commands unchanged.
- New `Destination` interface methods — additive; the only existing implementation (FS) gains them.
- `compile` skill text changes internally; slash-command surface (`/p-wiki:compile`, its `argument-hint`, its `allowed-tools`) is unchanged.
- `wiki-claude-md.template.md` gets two new bullets and one rewritten sentence; no schema or naming rule changes.

**No environment changes.** Node ≥ 18 was already required by v1.

**No schema changes.** Frontmatter, slug rules, link rules are byte-identical to v1.

**Existing wikis:** the first `compile` run after upgrade rewrites `index.md` with the new "Regenerated by `pwiki index`" marker line. This is a one-time cosmetic diff for users; nothing structural changes. Backlinks behave identically to current compile 4f (same regex, same skip-rules), so no spurious diffs are introduced there.

**Two known limitations for existing wikis:**

- `docs/wiki/CLAUDE.md` (copied from `wiki-claude-md.template.md` at `init` time) is **not** rewritten on plugin upgrade. Existing wikis keep the v1-era statement *"The CLI never touches body text"* in their auto-loaded LLM context, which is now mildly inaccurate. The CLI itself is unaffected; only the LLM-facing documentation is stale. Users can re-run `/p-wiki:init --force` (if/when supported) or copy the new template manually. We do not add an auto-migration step.
- `index.md` is fully overwritten on each `pwiki index` run. Manual edits to `index.md` are lost. This is identical to v1 compile step 5 behavior; we make it explicit here because the new marker line (`_Regenerated by `pwiki index`. Don't edit by hand._`) is intended to warn users.

**Version bump:** `v1.0.0` → `v1.1.0` (minor) per the rules in `.claude/CLAUDE.md`. Justified by additive scope and absence of breaking changes.

## 8. Release procedure

Per `.claude/CLAUDE.md` "Release tagging on git push":

1. Bump `plugins/p-wiki/.claude-plugin/plugin.json#version` from `1.0.0` to `1.1.0` in the same push that lands the implementation.
2. Propose the version and reasoning to the user: *"minor — adds `pwiki backlinks` and `pwiki index` commands and two methods to the Destination interface; nothing removed; no env requirement change."* Wait for explicit confirmation.
3. After confirmation: `git tag v1.1.0`, `git push`, `git push --tags`.

## 9. Implementation order

The implementation plan (writing-plans skill, follow-up session) will sequence the work, but the dependency order is:

1. `md.mjs` (`sanitize`) — pure helper, prerequisite for `index.mjs`.
2. `backlinks.mjs` (pure scan + match logic, no FS writes) + unit tests.
3. `index.mjs` (pure render + summary extraction, no FS writes) + unit tests.
4. Wire both into `destinations/fs.mjs` (`applyBacklinks`, `regenerateIndex`).
5. Add CLI dispatch in `pwiki.mjs` for the two commands.
6. Update `destination-contract.test.ts` (shape conformance).
7. Update `compile/SKILL.md` and `_shared/templates/wiki-claude-md.template.md`.
8. Extend `cli-integration.test.ts`.
9. Bump version, propose tag, release.
